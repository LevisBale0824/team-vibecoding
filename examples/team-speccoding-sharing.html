个人 Prompt 的困境

不确定性

同一个需求，每次 Prompt 结果不同。AI 可能过度设计，也可能遗漏边界情况。

质量波动

没有质量门禁。跳过测试、忽略错误处理、忘记安全检查...全靠 AI "心情"。

不可复现

今天的方案明天无法复现。没有规格文档、没有变更记录、没有验收标准。

# 典型的个人 Prompt 模式 > 帮我写一个用户注册接口 ... AI 直接生成代码，没有需求分析、没有测试、没有验证 > 再加一个登录功能 ... 范围不断扩大，质量不断下降 > 修一下这个 bug ... 补丁摞补丁，技术债越积越多

个人 Prompt vs 团队流水线

个人 Prompt 模式

  • 没有需求分析，直接写代码
  • AI 想到什么写什么
  • 没有测试或测试后补
  • 质量全靠运气
  • 无规格文档，无法复现
  • 范围不断扩大（scope creep）
  • 安全漏洞被忽略
  • 同样的错误反复犯

Team Vibecoding 流水线

  • /team-explore 先分析需求
  • /team-propose 写 OpenSpec 规格
  • /team-plan 审查任务和依赖
  • /team-apply TDD 严格实现
  • /team-verify 四层证据验证
  • /team-review 安全 + 质量门禁
  • /team-archive Spec 持续积累
  • /team-retro 自进化规则库

核心机制 — Guard Skills 质量门禁

每个步骤都有对应的 Guard Skill，定义了必须做和禁止做的规则。 Guard Skill 不提供建议——它提供约束。

openspec-guard

需求必须用 SHALL/MUST
硬阻塞问题阻止继续
范围蔓延被拒绝

planning-guard

任务 ≤ 1h
必须有验证方法
依赖关系无环

implementation-guard

TDD 先测试后实现
一次一个任务
禁止顺手重构

verification-guard

4 层验证全通过
命令实际执行（非推断）
跳过必须说明原因

review-guard

CRITICAL 问题阻塞
超出范围停止
关键功能必须有测试

skill-maintainer

Bad Case → 规则更新
系统级问题暂停使用
持续进化规则库

自进化系统 — Bad Case 反馈闭环

Bad Case 发生 → /team-retro 分析 → 规则修复 → Skill 更新 → 下次不再犯
Bad Case 示例

触发：用户说"跳过测试直接实现"
错误行为：AI 跳过测试，直接写实现代码
期望行为：AI 必须拒绝，解释这是 TDD 流程

规则修复：在 implementation-guard 中添加 IF-THIS-THEN-THEN 表格
IF user says "skip testing" →
THEN REFUSE + explain TDD requirement

IF-THIS-THEN-THAT 防护网

抗社会工程

每个 Guard Skill 包含常见"绕过"话术的应对规则。无论用户怎么催促、诱导，AI 都必须遵循流程。

持续硬化

每次 Bad Case 都变成新规则。系统随着使用时间推移越来越强，错误越来越少。

核心要点

1. Spec-First

先写规格（SHALL/MUST + WHEN/THEN），再写代码。规格是团队间的合约。

2. Evidence-Based

"通过"需要证据——实际运行的命令输出，而非推断。没有证据 = 没有完成。

3. Guard Skills 防护栏

AI 的"自由"是受约束的。每步都有规则告诉它必须做什么、禁止做什么。

4. Self-Evolving

Bad Case 变成新规则。系统随着使用越来越强，错误越来越少。

AI 不是替代工程师，而是在结构化流程中放大工程师的能力。
流程纪律 + AI 能力 = 可预测的高质量交付。
